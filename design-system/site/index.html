Principles

Comprehension over collection

Don’t show more data — fuse and rank it. Kill-chain, ATT&CK, and severity answer “what now?”.

Density with discipline

Pack information, but one idea per panel and a strict compact scale. Never noise.

Severity is the signal

Monochrome is the canvas; color is reserved for the 5-state severity model and selection.

Meaning survives without color

Every severity cue pairs hue with a glyph or label (1.4.1). Grayscale loses nothing.

Accessible by default

WCAG 2.2 AA is a gate, not a polish pass — keyboard, focus, targets, contrast.

Systematic

Every visual value traces to a token in app/globals.css, mirrored here and in Figma.

Foundations

Getting Started

How to consume the system in code and design, and the architecture that keeps both in sync.

Source of truth

The runtime source of truth is app/globals.css — CSS custom properties under :root (light) and [data-theme="dark"] (dark). The JSON in tokens/ and the Figma variables mirror it. Never hardcode a visual value; reference a token.

Using it in code

var(--color-ink)primary text / headings
var(--panel-bg)panel surface
var(--sev-critical-solid)critical badge fill (white text)
var(--spacing-md)12px gap/padding

Run npm run validate:theme to verify every token pair passes WCAG AA before shipping.

Using it in Figma

Four variable collections — Primitives, Scale, Type, Color. The Color collection carries three modes (Light / Dark / High Contrast); set a frame’s mode to theme it. Bind components to semantic variables (color/*, sev/*), never to primitives.

Token architecture

Three tiers: Primitives (raw hex/px) → Semantic (role aliases like color/bg/canvas, sev/critical/fill) → Component (bound on instances). Semantics alias primitives — never duplicate raw values.

Standards

Accessibility

Target WCAG 2.2 AA / AODA. The product is axe-clean across all 13 screens in both themes; npm run validate:theme gates every release.

The gates

  • Contrast (1.4.x): body ≥ 4.5:1, large/UI ≥ 3:1 — every token pair checked. Bright severity fills use -text variants when used as text.
  • Color independence (1.4.1): severity always pairs hue with a glyph or word — never a bare colored dot.
  • Keyboard (2.1): every control is a real <button>/<a>/input; clickable rows get tabindex + Enter/Space + label.
  • Focus (2.4): visible :focus-visible ring everywhere; the triage drawer is a real modal (trap + restore); off-canvas nav uses inert; a skip link is the first tab stop.
  • Targets & motion (2.5, 2.3): hit targets ≥ 24px (≥ 44px touch nav); prefers-reduced-motion honored.
  • Structure (1.3, 4.1): real heading order, list-based nav, th headers, icon-only controls labelled, informative SVGs get role="img".
This pageThis documentation portal follows the same rules: skip link, theme toggle with aria-pressed, keyboard-operable nav, reduced-motion support, and color-independent severity.

Standards

Governance

How the system changes, safely.

Contribution → release

  • Propose in docs/design-decisions.md (ADR style) — context, decision, consequences.
  • Review against .claude/rules/ (design-system, accessibility, code-style, content-voice).
  • Approve → implement on-system (token-bound, all states, accessible path).
  • Release after lint, build, validate:theme, and an axe pass are green.

Versioning (SemVer)
